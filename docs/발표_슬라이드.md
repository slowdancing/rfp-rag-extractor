# 입찰메이트 RFP RAG — 발표 슬라이드 대본

> PPT로 옮길 슬라이드별 구성. 각 슬라이드 = 제목 + 핵심 불릿(간결) + (말할 포인트).
> 흐름: **문제 → 핵심 아이디어 → 구현 → 실험 → 문제해결 → 배포·성과**

---

## [1] 표지
- **입찰메이트 RFP RAG Extractor**
- 복잡한 제안요청서(RFP)를 AI로 **추출·요약·추천·적격성 판정**
- 4팀 · B2G 입찰지원 컨설팅 엔지니어링

---

## [2] 문제 정의 — 왜 만들었나
- 나라장터에 **하루 수백 건 RFP**, 한 건당 수십 페이지(HWP/PDF)
- 컨설턴트가 일일이 못 읽음 → **요구조건·발주기관·예산·자격**만 빨리 알고 싶다
- "우리 회사가 이 사업 **할지 말지**"까지 판단 지원 필요
- 🎯 목표: **RFP 100건 기반 RAG 시스템** + 정량 실험으로 최적 구성 확정

---

## [3] 핵심 아이디어 (3줄)
1. **RAG** — 문서를 임베딩·검색해 근거로 LLM이 답 → 환각↓·출처 있음
2. **교체 가능한 설계** — 임베딩·LLM·벡터스토어를 추상화 → **`.env`만 바꿔** OpenAI ↔ 자체호스팅 전환
3. **자체호스팅** — 한국어 오픈모델(bge-m3·EXAONE)을 GCP VM에서 **무료·기밀보호**로 운영

> 말할 포인트: "OpenAI로 빠르게 개발 → 실험으로 검증 → 자체호스팅으로 갈아탐. 코드 변경 없이."

---

## [4] 시스템 한눈에 (아키텍처)
```
RFP zip → 추출·정제·청킹 → 임베딩 → ChromaDB
                                        │
질문/프로필 → 질의재작성 → 하이브리드 검색(BM25+dense) → LLM 재랭킹 → 답변/추천
                                        │
                         FastAPI(백엔드) ↔ React(웹 UI)
```
- 데이터 파이프라인 · RAG 엔진 · 웹 서비스 3부분
- 3단 추상화: `base(규격) + 구현체 + 팩토리`

---

## [5] 데이터 파이프라인 (+ 첫 문제)
- 입력: RFP 100건 (HWP 96 / PDF 4) + 메타데이터 CSV
- 흐름: `추출 → 정제 → 청킹 → 임베딩`
- ⚠️ **문제**: ① 깨진 문자(PUA·U+FFFD) 섞임 ② 일부 문서 텍스트가 89자로 잘림
- ✅ **해결**: ① 정제 로직으로 깨진 문자 제거 ② 짧은 문서 21건 **원본 재추출 보강**(89자→2만자)

---

## [6] 핵심 설계 ① — 교체 가능한 추상화
- 세 계층(임베딩·LLM·벡터스토어)에 동일 패턴: **base.py(ABC) + 구현체 + 팩토리**
- **Ollama = OpenAI 호환 API** → `provider=openai` + `base_url`만 바꾸면 자체호스팅 EXAONE도 **기존 코드 그대로**
- 효과: OpenAI/자체호스팅/HuggingFace를 **코드 0줄 수정**으로 비교·전환

---

## [7] 핵심 설계 ② — 하이브리드 검색
- ⚠️ **문제**: 자연어 질문("~사업규모와 과업기간은?")에 관련 문서가 상위에 안 잡힘
- 원인 규명(실험): **청크 크기 아님** → **질의 표현**이 문제 (일반어가 핵심어 희석)
- ✅ **해결**:
  - **질의 재작성** — 질문 → 핵심 키워드
  - **하이브리드 검색** — BM25(키워드) + dense(의미)를 **RRF로 융합**
  - → 실패 질의 **14위 밖 → 1위**

---

## [8] 실험 A — 검색 방식 (dense vs 하이브리드)
- 골든셋 285건, doc_id 정답 기준

| 방식 | Hit@1 | Hit@5 | MRR |
|------|------:|------:|------:|
| dense | 0.807 | 0.937 | 0.861 |
| **hybrid** | **0.933** | **0.975** | **0.953** |

- → 하이브리드가 **전 지표 우위**. "청크 크기 가설"은 실험으로 반증(256~1024 비슷).

---

## [9] 실험 B — 임베딩 모델 (자체 vs OpenAI)
| 모델 | 호스팅 | Hit@1(hybrid) | MRR |
|------|------|------:|------:|
| text-embedding-3-small | OpenAI | 0.937 | 0.955 |
| **bge-m3** | 자체(무료) | **0.961** | **0.981** |
| KURE-v1 | 자체(무료) | 0.958 | 0.978 |

- **한국어 데이터에선 한국어 오픈모델이 OpenAI 범용 임베딩을 능가**(dense 격차 +0.13)
- 무료·자체호스팅인데 품질도↑ → **임베딩 = bge-m3 채택**

---

## [10] 실험 C — LLM (자체 EXAONE vs OpenAI)  ★핵심
- 검색 고정(bge-m3), LLM만 교체. 골든셋 + **LLM-judge 교차검증**(심판 2개)

| 비교축 | EXAONE-3.5-7.8B(자체) | gpt-5-mini(OpenAI) |
|------|:---:|:---:|
| 추출 정확도 | 0.533 | 0.533 (**동률**) |
| 서술형 답변품질(judge) | **4.37** | 4.20 |
| 응답속도(중앙값) | **3.78s** | 9.81s |
| 비용·기밀보호 | ✅ 무료·내부 | 유료·외부 |

- 상대 심판 gpt조차 **EXAONE(4.31) > 자기(4.14)** → 자기편향 아님, 견고
- → **정확도 대등 + 품질·속도(2.6배)·비용·보안 우위** → **LLM = EXAONE 채택**
- (참고) Qwen2.5는 한국어 답변에 중국어 혼입 → 부적합

## [13] 배포 — 자체호스팅 (+ 문제해결)
- GCP VM(L4) + **Ollama**로 EXAONE·bge-m3 상시구동 + FastAPI
- ⚠️ **문제들 & 해결**:
  - sudo 없음 → Ollama 홈 설치, pip 대신 표준 라이브러리
  - 외부에서 VM 호출 불가(방화벽) → **GCP 방화벽 포트 개방(내 IP)** + `0.0.0.0` 바인딩
  - 웹전용 환경 → **노트북 라이브 데모**도 준비
- ✅ 결과: **로컬 브라우저 → VM의 EXAONE 서비스** 실동작

## [15] 트러블슈팅 하이라이트 (개발 중 문제→해결)
| 문제 | 원인 | 해결 |
|------|------|------|
| AI 요약이 줄글·헛소리 | EXAONE 형식 이탈, 온도 0.8 | few-shot 예시 + **온도 0.2** + 파서 강건화 |
| 긴 문서 요약 항목 빔 | 사업개요 청크 놓침 | 핵심 항목 **메타데이터로 직접 채움** |
| "웹사이트" 추천 0건 | 관련성 게이트가 웹≠홈페이지 오판 | 게이트 완화(분야만 관련되면 통과) |
| VM 코드 반영 안 됨 | 옛 프로세스 미종료 | `git reset --hard` + 프로세스 확실히 재시작 |

> 교훈: LLM 형식 문제는 **온도↓ + few-shot + 규칙기반 보정**, 배포 문제는 **반영 확인**이 핵심
