# 생성 평가 결과

RFP 질문에 대한 LLM 답변 품질을 두 가지 골든셋으로 평가한다.

---

## ① 메타데이터 골든셋 — 정답 포함 정확도 (contains-matching)

- 평가셋: `data/eval/eval_set.draft.jsonl` (표본 30건, 자동 생성·doc_id/메타 기반)
- 채점: 정답 문자열(예산·마감·발주기관)이 답변에 포함되는지 (`is_correct`)

| 유형 | 정확도 | 정답/전체 |
|------|------:|------:|
| 전체 | 0.500 | 15/30 |
| budget | 0.533 | 8/15 |
| deadline | 0.000 | 0/8 |
| agency | 1.000 | 7/7 |

> deadline 0.000은 채점 방식 한계: 정답이 `2026-03-15`인데 답변은 "3월 15일" 등
> 표기가 달라 문자열 불일치. 실제 오답이 아니라 **정확 문자열 매칭의 과소평가**.
> 발주기관(agency)은 표기가 고정적이라 1.000.

---

## ② 내용형 골든셋 — LLM-judge (답변 품질 점수)

- 평가셋: `data/eval/eval_set.content.draft.jsonl` (28건, 본문 근거 질문 — LLM 생성 10건 + 직접 작성 18건, 원문 대조)
- 채점: LLM-as-judge 5점 척도 (근거 부합·정확성), 검색은 Hit@5 병행 측정

| 지표 | 값 |
|------|---:|
| 검색 Hit@5 | **0.96** |
| 답변 품질 평균 | **3.89 / 5** |

> 본문 근거가 필요한 질문에 대해 대체로 정확히 답변. 검색이 상위 5 안에 정답
> 문서를 96% 포함 → 검색은 안정적이며, 남은 개선 여지는 답변 생성(LLM) 쪽.
> 표기 불일치에 강건한 LLM-judge라 ①의 과소평가를 보완하는 지표.

## 재현
```bash
# 메타데이터 기반 (①)
python -m scripts.eval_generation
# 내용형 LLM-judge (②)  — 골든셋 eval_set.content.draft.jsonl 사용
```
